| 主线 | 能力 | 实战对应 |

## 🎯 实战目标

三步串成一条完整的中医药 AI 研究管线：

```
期刊 OCR txt ──→ 抽医案存 CSV ──→ 嵌入建库 ──→ 流式问答前端
                                              └─→ 多智能体辨证
```

**关键原则**：
- ✅ **全自己写，不套框架** —— 不用 LangChain / AutoGen / ChromaDB
- ✅ **零配置向量库** —— 纯 numpy 实现，pickle 存储
- ✅ **可复现** —— 完整的数据处理管线 + 代码 + 结果

## 🚀 快速开始

### 环境准备

```bash
# 创建虚拟环境
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate

# 安装依赖
pip install -U openai numpy pandas streamlit tqdm

# 设置环境变量（通义百炼 OpenAI 兼容接口）
export DASHSCOPE_API_KEY="<你的百炼 API Key>"
# base_url: https://dashscope.aliyuncs.com/compatible-mode/v1
```

### 第一步：医案抽取（extract_cases.py）

从约 100 篇中医期刊 OCR 文本中抽取结构化医案：

```bash
python extract_cases.py
```

**技术要点**：
- 异步并发处理（Semaphore 控并发 30）
- JSONL 断点续传（故障恢复）
- OCR 文本预清洗（全角转半角、断行合并）
- tqdm 进度条可视化

**产出**：`output/medical_cases.csv`（113 则医案）

### 第二步：构建向量知识库（build_kb.py）

将医案切块并向量化，构建本地知识库：

```bash
python build_kb.py
```

**技术要点**：
- 语义切块（>800字自动切分，带重叠）
- 批量嵌入（10条/批，并发 8）
- L2 归一化（点积 = 余弦相似度）
- pickle 存储（零配置）

**产出**：`output/kb.pkl`（232 文本块，1024维向量）

### 第三步：启动应用

**RAG 问答系统**（端口 8501）：
```bash
streamlit run app.py
```

**多智能体辨证系统**（端口 8502）：
```bash
streamlit run agents_app.py --server.port 8502
```

### 2. 多智能体辨证论治

**协作流程**：
```
🔍 检索Agent → 🩺 辨证Agent → 💊 处方Agent → ✅ 审校Agent
     ↑                                              │
     └────────── 审校不通过，带意见回炉 ────────────┘
                    （最多 2 轮）
```

**四个 Agent 角色**：

| Agent | 图标 | 职责 | 输出 |
|-------|:----:|------|------|
| 检索Agent | 🔍 | 从知识库检索相似医案 | 关键症状、相似医案、参考意见 |
| 辨证Agent | 🩺 | 中医辨证分析 | 证型、病因、病机、病位 |
| 处方Agent | 💊 | 制定治疗方案 | 治法、方剂、药物组成 |
| 审校Agent | ✅ | 审核方案合理性 | 一致性检查、安全性检查 |

## 🧠 技术选型思考

### 为什么不用知识图谱？

指南中明确阐述了**知识图谱退出主路径的五个根本原因**：

1. **把动态语境压成静态关系** —— 「口干」可见于阴虚/湿热/阳明热盛等不同语境，图谱会制造假确定性
2. **构建过程逐环引入噪声** —— 实体抽取→关系抽取→社区发现，每多一环就多一个误差源
3. **"可解释性"常是形式解释** —— 症状→证候→治法的路径答不了「证据出自哪条」
4. **创新空间已窄** —— 本体/图嵌入/GraphRAG 等变体已拥挤，边际提升有限
5. **更新机制与知识增长不匹配** —— 图谱每次更新都涉及冲突消解/版本追踪/专家核验

**替代方案**：向量库 + 混合检索 + 重排序 + LLM 抽取 + Agent + 规则 + 专家评价

### 为什么全自己写？

> 仅用现成工程工具（如 Dify / RAGFlow / FastGPT）拼装，能快出演示，却做不出有创新、有论证的科研。

本项目的检索、拼 prompt、Agent 编排全部手写，目的是：
- 理解每一步的原理
- 便于调试和改进
- 科研可复现
